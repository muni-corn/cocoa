# generating changelogs

i can automatically generate changelogs for you based on your commit history! they will be almost as
beautiful as i am~

## what is a changelog?

changelogs document changes in each release:

```markdown
# Changelog

## [1.2.0] - 2024-03-15

### Features

- auth: add password reset functionality
- auth: implement OAuth 2.0 support
- dashboard: add user analytics dashboard

### Bug fixes

- core: fix memory leak in parser
- fix: prevent duplicate form submission

### Performance

- db: optimize database queries with indexing

### Breaking changes

- authentication endpoints now require 2FA setup

## [1.1.0] - 2024-02-10

### Features

- auth: add two-factor authentication
- notifications: add email notifications

### Bug fixes

- utils: fix timezone handling

---
```

voila! all generated by yours truly~

## generating a changelog

### generate for unreleased changes

```bash
cocoa changelog
```

This generates changes since the last version tag:

```markdown
# Changelog

## Next version

### Features

- auth: add password reset
- auth: implement SSO

### Bug fixes

- auth: fix login timeout
```

### generate for a specific commit range

```bash
cocoa changelog v1.0.0...HEAD

# or by date
cocoa changelog --since "2024-01-01"

# or last n commits
cocoa changelog --last 20
```

### full changelog from everything

```bash
cocoa changelog --all
```

generates entire history from first commit to now.

## configuration

customize changelog generation in `.cocoa.toml`:

```toml
[changelog]
output_file = "CHANGELOG.md"
include_merge_commits = false
include_reverts = true
date_format = "%Y-%m-%d"

[changelog.sections]
feat = "Features"
fix = "Bug fixes"
perf = "Performance"
docs = "Documentation"
refactor = "Refactoring"
style = "Style"
test = "Tests"
chore = "Maintenance"
breaking = "Breaking changes"
```

### key settings

| Setting                 | Default        | Meaning                  |
| ----------------------- | -------------- | ------------------------ |
| `output_file`           | `CHANGELOG.md` | Where to write changelog |
| `include_merge_commits` | `false`        | Include merge commits    |
| `include_reverts`       | `true`         | Include revert commits   |
| `date_format`           | `%Y-%m-%d`     | Version date format      |

### custom section names

```toml
[changelog.sections]
feat = "✨ New features"
fix = "🐛 Bug fixes"
perf = "⚡ Performance"
docs = "📚 Documentation"
breaking = "💥 Breaking changes"
```

now i can make your changelog even nicer!

## real-world examples

### example 1: simple feature release

commits since last version:

```
feat(auth): add password reset
feat(auth): add two-factor authentication
fix(auth): prevent concurrent logins
docs: update security docs
```

Generated changelog:

```markdown
## [1.1.0] - 2024-03-15

### Features

- auth: add password reset
- auth: add two-factor authentication

### Bug Fixes

- auth: prevent concurrent logins

### Documentation

- update security docs
```

### Example 2: Major Release with Breaking Changes

Commits:

```
feat(api): redesign authentication endpoints
feat(api): add rate limiting
fix(api): prevent slow requests
docs: update API documentation

BREAKING CHANGE: Old /auth endpoints removed
```

Generated changelog:

```markdown
## [2.0.0] - 2024-03-15

### Breaking changes

- Old /auth endpoints removed

### Features

- api: redesign authentication endpoints
- api: add rate limiting

### Bug fixes

- api: Prevent slow requests

### Documentation

- docs: update API documentation
```

### example 3: emergency hotfix release

Just one fix:

```
fix(security): patch authentication bypass vulnerability
```

Generated changelog:

```markdown
## [1.0.1] - 2024-03-14

### Bug fixes

- security: Patch authentication bypass vulnerability
```

## Writing Changelog-Friendly Commits

### Good Changelog Commits

these look great in changelogs:

```
feat(auth): add password reset
# clear, specific action :D

feat(dashboard): add user analytics
# specific feature :o

fix(ui): prevent duplicate form submission
# clear problem being fixed ^o^
```

### Poor Changelog Commits

these don't read well:

```
feat: stuff
# too vague :(

feat(auth): work in progress
# what kind of feature is that: >:T

feat(auth): add authentication
# okay? too generic though :c

feat(auth): add password reset and 2FA and OAuth and account linking
# consider breaking up your commits! </3
```

## changelog-specific footers

you can use footers to refine changelog:

```
feat(api): add rate limiting

Implement token bucket algorithm for rate limiting.
All endpoints now support rate limiting headers.

Refs #123          # Reference without closing
Closes #123        # Auto-close issue
Closes #123, #124

BREAKING CHANGE: Old rate-limit header renamed to X-Rate-Limit
```

in changelog:

```markdown
### Features

- api: add rate limiting

### Breaking changes

- Old rate-limit header renamed to X-Rate-Limit
```

## updating files in changelog

i can automatically update version files for you!

```toml
[version]
commit_version_files = [
  "package.json",
  "Cargo.toml",
  "pyproject.toml",
  "src/version.rs"
]
```

Example automatic update:

**before (package.json):**

```json
{
  "version": "1.0.0"
}
```

**after running release:**

```json
{
  "version": "1.1.0"
}
```

## integration with releases

remember that this:

```bash
cocoa release
```

... is the same as this:

```bash
cocoa bump auto  # analyzes commits, bumps version (1.0.0 → 1.1.0)
cocoa changelog  # generates changelog
cocoa tag        # creates git tag with changelog
```

then, just push everything to your git repository:

```bash
git push --all --tags
```

this does everything:

1. ✅ Bumps version (analyzes commits)
2. ✅ Updates version files
3. ✅ Generates changelog
4. ✅ Creates git tag
5. ✅ Commits changelog
6. ✅ Pushes to remote

See [Version Management](./versioning.md) for details.

## tips and tricks

### 1. exclude noise

you can exclude some commit types completely:

```toml
[changelog]
# don't include these types
exclude_types = ["chore", "style", "test"]
```

### 2. milestone-based changelogs

for long releases, you can organize changelogs by milestones:

```bash
cocoa changelog v1.1.0...v1.2.0 > releases/v1.2.0.md
cocoa changelog v1.0.0...v1.1.0 > releases/v1.1.0.md
```

then create one root CHANGELOG.md that includes them.

### 3. highlight important changes

use breaking changes:

```
feat(auth): redesign login flow

Now supports OAuth, SAML, and WebAuthn in addition to
username/password authentication.

BREAKING CHANGE: Password complexity requirements changed
BREAKING CHANGE: Login endpoint response format changed
```

i'll always put breaking changes first!

## troubleshooting

### "no commits since last version"

this means i don't see any commits made since your last release.

make new commits, then try again.

### "changelog looks empty or incomplete"

you should check that your configuration matches your commits:

```bash
# list recent commits with their types
git log --oneline | head -20

# see what i see
cocoa changelog --verbose
```

also, do your configured section names match your commit types?

```toml
# Wrong
[changelog.sections]
features = "Features"  # section name is singular
feature = "Features"   # commit type is singular, but mismatch

# Correct
[changelog.sections]
feat = "Features"      # match commit type exactly
fix = "Bug fixes"
```

### "changelog not updating"

here's a checklist:

1. have you added `output_file = "CHANGELOG.md"` to `.cocoa.toml`?
2. have you made commits with types in `[changelog.sections]`?
3. did you create git tags for past versions?
4. have you excluded any types you care about?
